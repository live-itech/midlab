API Documentation

Spesifikasi API bridging MidLab dengan Laboratory Information System (LIS)

Dokumen ini mendeskripsikan dua arah API antara MidLab dan LIS:

  • Order API — LIS POST order pemeriksaan ke MidLab (inbound).
  • Result API — MidLab POST hasil pemeriksaan ke LIS (outbound).

Untuk konfigurasi URL & API key, gunakan halaman Settings. URL Order API otomatis mengikuti IP server.

1. Order API — LIS → MidLab

Endpoint yang harus dipanggil LIS untuk mengirim order pemeriksaan ke alat lab.

URL di atas otomatis mengikuti IP server MidLab. Jangan hardcode di sisi LIS — selalu konfirmasi via halaman Settings.

1.2 Authentication

Jika order_receiver.api_key di-set di /etc/midlab/config.yaml, setiap request wajib menyertakan header X-API-Key: <key>. Tanpa key yang benar, server merespon 401 Unauthorized. Jika config kosong, autentikasi di-skip (development mode).

1.3 Endpoints

1.4 Status Order

  • pending — Tersimpan, menunggu dikirim ke alat oleh TCPSocketService.
  • sent — Sudah terkirim ke alat (alat menerima order).
  • failed — Gagal kirim. Cek failed_at_service dan error_message.

2. Result API — MidLab → LIS

MidLab akan POST hasil pemeriksaan ke endpoint LIS sesuai konfigurasi di halaman Settings. Tim LIS perlu menyediakan endpoint yang menerima struktur berikut.

2.1 Kontrak Endpoint LIS

LIS harus menyediakan endpoint POST <your-url> yang:

  • Menerima Content-Type: application/json.
  • Menerima header X-API-Key: <key> jika di-set di MidLab.
  • Return 2xx bila result diterima & valid → MidLab tandai send_status='sent'.
  • Return 4xx/5xx bila tolak → MidLab tandai send_status='failed' dan retry sesuai retry_max di config (default 3x).

2.2 Schema Result (yang MidLab kirim)

{
  "mid_version": "1.0",
  "instrument_id": 1,
  "protocol": "ASTM",                  // ASTM / HL7 / COBAS_C111
  "message_id": "uuid-v4",
  "message_datetime": "2026-05-13T10:35:22+00:00",
  "patient": {
    "patient_id": "PAT001",
    "name": "John Doe",
    "dob": "19900515",
    "gender": "M",
    "physician": "Dr. Smith"
  },
  "specimen": {
    "sample_id": "SAMP001",
    "sample_type": "Blood",
    "collected_at": "20260513093000"
  },
  "order": {
    "order_id": "ORD-12345",
    "panel": "CBC"
  },
  "results": [
    {
      "test_code": "WBC",
      "test_name": "White Blood Cell",
      "value": "5.2",
      "unit": "10^3/uL",
      "reference_range": "4.0-10.0",
      "flag": "N",                     // N / H / L / A / <  / >
      "status": "F"                    // F=Final, P=Preliminary, C=Correction
    }
  ],
  "comments": [
    "result[WBC]: Sol1: F Dev"
  ],
  "parse_errors": []                   // kosong kalau parsing bersih
}

2.3 Retry & Error Handling

  • Default poll interval 5 detik (configurable di config.yaml: result_sender.poll_interval).
  • Default retry_max=3. Setelah melebihi, result di-skip + log WARNING. Bisa di-retry manual dari halaman Results.
  • Connection error / timeout / non-2xx semua dianggap failed — pesan error disimpan di tbl_result.error_message.

2.4 Contoh Endpoint LIS (referensi)

# LIS endpoint (pseudo-code)
@POST /lis/api/results
def receive_result(body, headers):
    if headers.get('X-API-Key') != EXPECTED_KEY:
        return 401, {"error": "invalid api key"}
    try:
        lab_id = save_to_lis_db(body)
        return 200, {"success": True, "lis_id": lab_id}
    except ValidationError as e:
        return 400, {"error": str(e)}
    except Exception:
        return 500, {"error": "internal"}

3. Settings API — Internal

API untuk Web Console mengelola konfigurasi bridging — biasanya tidak dipakai langsung dari LIS.

GET /api/settings — Ambil semua setting + auto-detect IP.

PUT /api/settings — Update LIS URL/key.

POST /api/settings/test-lis — Test koneksi ke LIS.

4. Catatan Keamanan

  • Selalu gunakan X-API-Key di production — set order_receiver.api_key di config.yaml.
  • Untuk Result API: gunakan https:// agar credential tidak bocor di network.
  • Pasang reverse proxy (nginx) di depan port 8000 dan 8001 untuk TLS termination.
  • Restrict akses port 8001 hanya dari IP LIS via firewall.

5. Versioning

Field mid_version di setiap payload (request & response) menunjukkan versi schema MidLab. Saat ini "1.0". Perubahan breaking akan bump ke "2.0" dengan endpoint baru (/v2/api/...) — versi lama tetap di-maintain minimal 6 bulan.
